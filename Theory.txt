Структура проєкту:
- node_modules - пакети, залежності за допомогою яких ми запускаємо тести (створюються за допомогою npm команди).Якщо їх видалити - знову з'являться після команди npm install
- package.json - показує що це за проєкт, його мапа(паспорт проєкту)
* evDependencies - версії пакетів із node_modules
* scripts - коротка команда замість назви скрипта
* config файл далі запускає наші тести
- test => тут ми прописуємо наші тести, в блоках describe та it
- package-lock.json = потрібен щоб Dependencies спрацювали в будь якому випадку, версії підібрані правильно

- wdio.conf.js => файл конфігурації для запуску наших тестів, як вони повинні запуститись:
* runner - можна запускати локально та у хмарних сервісах
* specs - які саме файли ми хочемо запускати (**/*.js - всі файли JS)
* exclude - які файли ми не будемо запускати
* maxInstances - максимальне число браузерів, яке можна запустити одночасно
* capabilities - щоб вказати, на якому браузері ми запустимо автотести, що пропускати
* logLevel - все, що я бачу у консолі(логи)
* baseUrl - URL, з якого ми будемо запускати наші тести
* waitforTimeout - кількість секунд чекати на елемент
* connectionRetryTimeout - перепідключення 2 хв
* connectionRetryCount - 3 рази спробує перепідключитися
* services - при роботі з хмарними сервісами(browserstack)
* framework - який фреймворк використовується для тестів
* reporters: ["spec"] - вивід результату саме в командний рядок
* mochaOpts - стиль написання тесту - bdd - behavior driven development (як користувач)
* Hooks - before - щоб створити тестові дані перед автотестами, щоб скоротити код, after - якщо треба видалити якісь дані
